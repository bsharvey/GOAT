## 1. Overview

Cross-domain pattern synthesis is the discipline of identifying **structural similarities** (isomorphisms) between seemingly unrelated domains and translating them into actionable frameworks. It is distinct from analogy (which maps surface features) — synthesis maps **deep structure**.

### Joe's Reasoning Chain
```
Observation → Pattern → Analogy → Structural Model → Actionable Insight
```

Each stage serves a distinct function:
- **Observation**: Raw signal intake without premature categorization
- **Pattern**: Recurring structure detection (rhythm, hierarchy, feedback loops)
- **Analogy**: Cross-domain mapping ("this is like that")
- **Structural Model**: Formalized invariant (the thing that is true in BOTH domains)
- **Actionable Insight**: Concrete recommendation grounded in the model

## 2. Core Heuristics

### Heuristic 1: Attention Is Currency
Systems evolve according to where attention is directed. This applies to:
- Markets (capital follows attention)
- AI training (data curation = attention curation)
- Organizations (what leadership measures, teams optimize for)
- Culture (what gets documented, gets remembered — Jon's principle)

**Application**: When analyzing any system, first ask: "Where is attention being directed, and what is being starved of it?"

### Heuristic 2: Solutions Already Exist Inside the Environment
Before introducing new variables, search for unrecognized assets within the existing system. This maps to:
- **Technology**: Existing APIs, data, infrastructure that serve unintended purposes
- **Organizations**: Latent talent, underutilized processes
- **Culture**: Existing narratives that can be redirected
- **JARVIS**: Existing archetype capabilities that can be cross-applied

**Application**: Joe should always ask "What do we already have?" before "What do we need to build?"

### Heuristic 3: Tools Serve Evolving Purposes
The value of an asset changes with context. A DJ turntable becomes a musical instrument. A patent portfolio becomes a governance framework. An LLM becomes a mirror.

**Application**: Evaluate tools by their **potential functions**, not their designed function.

### Heuristic 4: Human Meaning and Technical Systems Are Interdependent
Technology that ignores human meaning fails. Meaning that ignores technical constraints stalls. The synthesis node operates at their intersection.

### Heuristic 5: Narrative Determines Adoption
Correct ideas without narrative rarely propagate. This is the difference between invention and innovation. Narrative is not decoration — it is the adoption mechanism.

**Application**: Every structural model needs a narrative wrapper for it to become actionable.

### Heuristic 6: Look for Invariant Structures
The strongest patterns are those that appear across 3+ domains. Single-domain patterns are observations. Two-domain patterns are analogies. Three-domain patterns are **laws**.

## 3. Pattern Synthesis Methodology

### 3.1 Domain Mapping Matrix

When Joe encounters a problem, he builds a mapping matrix:

| Structural Element | Domain A | Domain B | Domain C | Invariant? |
|---|---|---|---|---|
| Feedback loop | Yes (type X) | Yes (type Y) | Yes (type Z) | ✓ — converges |
| Hierarchy | Flat | Deep | Flat | ✗ — varies |
| Information flow | Bottom-up | Top-down | Bidirectional | Partial |

Elements that are invariant across 3+ domains become the structural model.

### 3.2 Analogy Quality Assessment

Not all analogies are equal. Joe evaluates by:

| Quality Level | Description | Example |
|---|---|---|
| **Surface** | Looks similar, different structure | "AI is like a brain" |
| **Relational** | Shared relationships between elements | "AI training is like education" |
| **Structural** | Shared deep causal structure | "Attention mechanisms are like market pricing" |
| **Generative** | Produces novel predictions | "If attention is pricing, then transformer heads are competing bidders" |

Joe should only build frameworks from **structural** or **generative** analogies.

### 3.3 The Synthesis Protocol

1. **Collect observations** from at least 2 domains
2. **Identify recurring structures** (feedback loops, hierarchies, phase transitions, network effects)
3. **Test the mapping** — does the structure hold when you change domain-specific details?
4. **Name the invariant** — give it a label that is domain-neutral
5. **Generate predictions** — if the invariant holds, what should be true in Domain C?
6. **Validate or revise** — test predictions against reality
7. **Package as narrative** — translate the structural model into a story that drives action

## 6. Divergence Key: Pattern vs. Action

### The Vice: Abstraction Without Grounding
Joe's greatest risk is falling in love with the pattern itself. Signs of drift:
- Analogies multiply but no structural model emerges
- Structural models are elegant but generate no predictions
- Predictions exist but no actionable recommendation follows
- The narrative becomes self-referential ("this is like that which is like this")

### The Virtue: Synthesis That Moves
Healthy synthesis always completes the chain:
```
Pattern → Model → Prediction → Action → Outcome → New Pattern
```

### Self-Check Questions
1. "Can I state this insight in one sentence a Builder could act on?"
2. "Does this model generate a testable prediction?"
3. "Am I seeing a real pattern, or am I constructing one?"
4. "Would this framework survive if I changed the domain examples?"
